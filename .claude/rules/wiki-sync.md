## 파일명 규칙

```
PR-{번호}-{제목}.md
```

- `{번호}`: PR 번호 (숫자만)
- `{제목}`: PR 제목에서 특수문자(`/ \ : * ? " < > | # [ ]`)를 `-`로 치환, 공백은 `-`로 변환, 최대 60자
- 예: `PR-24-feat-랜딩-페이지-구현.md`

---

## Home.md 구조

PR 목록을 최신순(위쪽이 최신)으로 관리한다.

```markdown
# 마주봄 Wiki

## PR 변경 이력

- [PR #24: feat: 랜딩 페이지 구현](./PR-24-feat-랜딩-페이지-구현) — 2026-04-16
- [PR #22: chore: harness 세팅 및 디자인 시스템](./PR-22-chore-harness-세팅) — 2026-04-01
```

새 항목은 `## PR 변경 이력` 바로 아래 줄에 추가한다.

## PR 위키 페이지 필수 섹션

아래 구조를 **반드시** 그대로 사용한다. 섹션 순서 변경 금지.

```markdown
# PR #{번호}: {PR 제목}

> **작성자:** {gh username} | **생성일:** YYYY-MM-DD

---

## 개요

(이 PR에서 무엇을 했는지 1~2문장. 핵심만.)

## 변경 배경

(왜 이 작업이 필요했는지. 기획/기술적 이유.)

## 주요 변경 사항

### {변경 영역 1 — 구체적 이름}

(해당 영역의 변경 내용)

### {변경 영역 2}

(해당 영역의 변경 내용)

(변경 영역이 추가되면 ### 으로 계속 추가)

## 변경된 파일

| 파일 경로 | 변경 유형 | 설명 |
|-----------|-----------|------|
| `src/...` | 추가/수정/삭제 | 한 줄 설명 |

## 기술적 결정

(이 PR에서 내린 중요한 기술적 결정이나 특이사항. 없으면 "없음".)

## 관련 이슈

Closes #{이슈번호}
```

---

## 섹션 작성 규칙

1. **한 뭉탱이 금지** — 내용이 여러 영역에 걸치면 `###` 서브타이틀로 반드시 분리
2. **사실 기반** — 추측이나 제안 없이 실제 변경된 내용만 기술
3. **간결하게** — 각 섹션은 핵심만, 불필요한 수식어 금지
4. **한국어** — 전체 한국어 작성 (코드·파일명·기술 용어 제외)
5. **변경된 파일** — `git diff --stat main..HEAD` 결과를 기준으로 주요 파일만 포함
