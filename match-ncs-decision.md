# summary_text 안정성 개선 — NCS 매핑 정밀화 + temperature 조정

> PR 설명용 결정 근거 문서. 작업 후 PR 본문 작성 시 참고.

---

## 1. 배경

`POST /api/match`는 두 번의 OpenAI 호출로 매칭 결과를 만든다.

1. **1차** (`matchStrategy.buildMessages`): 프로필 → `radar_chart` + `top3_jobs` 선정 (화이트리스트 45개 직종 안에서)
2. **2차** (`buildSummaryText`): top3 직종명을 워크넷 NCS API에 검색 → 매칭된 NCS 키워드(능력단위·지식기술태도)로 한 문장 summary 생성

#199 리팩토링의 의도는 "summary가 NCS 키워드를 반드시 1개 이상 자연어로 녹이도록" 하는 것이었으나, 운영 중 다음 문제가 관찰되었다.

### 문제 1. NCS 매칭 누락이 잦음

화이트리스트 45개 중 NCS 검색어가 매핑된 직종은 16개뿐(`JOB_SEARCH_MAP`). 나머지 29개는 직종명 그대로 검색되며 NCS 결과가 0건인 경우가 많다.

검증 사례 (user_id=108, 바나나 유저):
- top3 = 점역·교정사 / 사무 보조 / 온라인 콘텐츠 모니터링
- NCS 매칭: 사무 보조 1건만 잡힘 (3건 NCS 데이터)
- 점역·교정사, 온라인 콘텐츠 모니터링은 0건 → summary가 사실상 "사무 보조" 키워드로만 생성됨

### 문제 2. 응답이 매번 달라짐

`openAiFetch.ts`의 `temperature: 0.7` + JSON mode 조합에서, 동일 입력에도 호출마다 표현이 크게 달라진다.

| 호출 | summary_text |
|---|---|
| DB 저장값 (이전 시점) | "정해진 패턴의 반복 작업에 집중하실 때 안정적이에요" |
| 시뮬레이션 (현재 코드 + 같은 데이터) | "PC와 데이터베이스 프로그램을 활용한 사무행정에 강점이 있어요" |
| radar 폴백 (repetition 최고) | "정해진 순서를 차분하게 따라가는 꼼꼼함이 큰 강점이에요" |

> 참고: DB의 기존 값이 NCS 키워드를 안 담은 이유는 **`buildSummaryText.ts`가 2026-05-10에 도입되었고, 해당 record의 마지막 갱신은 2026-04-25이기 때문**이다. #199 이전 코드(1차 호출에서 한 번에 summary까지 생성, NCS 미사용) 시절의 결과가 남아있는 것. 새로 호출되면 NCS 기반 summary로 덮어쓰여진다.

## 2. 분석 — NCS API 매핑 가능성 조사

화이트리스트 45개 각각에 대해 직종명·키워드 후보를 NCS API로 직접 검색하여 매칭 품질을 측정했다.

### 검색 결과 (40개 측정, 5개는 검색 후보 미수립)

| 구분 | 직종 수 | 비고 |
|---|---|---|
| 검색 결과 있음 | 39 | 형식적으로 NCS unit이 반환됨 |
| 검색 실패 | 1 | 재래시장관리 |

### 매칭 품질 분류

겉으로는 "39/40 성공"이지만, **실제 의미적 정확도**로 다시 분류하면:

#### ✓ 정확 매칭 (NCS unit이 직종 의미와 일치, ~10개)
- 사무 보조 → 사무행정
- 도서관 사서보조 / 대형서점 도서정리 → 도서관·문헌정보관리
- 정리 및 수납 → 정리수납·가사지원
- 환경정리 → 청소·환경미화
- 세탁물 관리 → 세탁
- 급식지원 → 급식조리·한식조리
- 부품 조립 → 조립·반도체장비·철도차량제작
- 제품 검수 보조 → 검수·검정
- 제과제빵 보조 → 제빵
- 영유아 돌봄 → 보육
- 실버케어 → 요양지원
- 반려동물돌봄 → 동물보건
- 농업·임업·어업관련 직무 → 농업
- 환자이송보조 → 환자이송·구조구급
- 홀몸어르신 안부확인 → 사회복지면담·요양지원

#### ⚠ 부정확 매칭 (NCS는 반환되지만 의미가 다른 분야)
- 안마사 → "안마" 검색 → 결과는 **아이돌봄, 전기설비설계**
- 헬스키퍼 → "안마" → 동일
- 점역·교정사 → "교정" → **편집, 프리프레스**
- 호텔객실관리 → "호텔객실" → **카지노운영, 크루즈운영**
- 방역·소독활동 → "방역" → **내수면양식, 수산질병관리**
- 온라인 콘텐츠 모니터링 → "콘텐츠" → **VR콘텐츠디자인, 애니메이션콘텐츠유통**
- 무인정보단말기(키오스크)안내 → "키오스크" → **국내여행안내, 이러닝콘텐츠개발**
- 기타: 우편물 분류, 문서파기, 기부물품관리, 대형마트 매장정리, 장난감 세척, 다회용품 세척, 이동보조기기 분해 세척 및 소독, 버스청결관리, 교통약자승하차지원, 간식 포장 및 점검, 은행서비스안내, 건강검진센터지원, 스포츠이용시설안내, 장애인전문주차 구역계도 및 홍보, 업사이클링·리사이클링 등

→ NCS 표준직무기술서 API의 검색은 키워드 부분 매칭이라 노이즈가 많다. 잘못된 매칭을 그대로 쓰면 summary가 "안마사 사용자에게 전기설비설계 키워드"를 녹이는 부작용이 생긴다.

#### ✗ 검색 실패
- 재래시장관리

## 4. 결정 — 옵션 D

### 4-1. JOB_SEARCH_MAP 정리 원칙
- **추가**: 검증된 정확 매칭 (영유아 돌봄→보육, 실버케어→요양, 반려동물돌봄→반려동물, 제품 검수 보조→검수, 농업·임업·어업관련 직무→농업, 홀몸어르신 안부확인→안부확인 등)
- **제거**: 의미가 명백히 다른 분야로 빠지는 매핑 (호텔객실→카지노, 이동보조기기 소독→가금사육 등)
- **유지**: 부분 매칭이지만 의미가 통하는 매핑 (공공자전거세척→자전거정비 등)

### 4-2. NCS 0건 직종 폴백 강화
- 현재 `buildRadarFallbackSummary`는 radar 최고축 1개 기준 5개 고정 문장 → **매번 동일하고, top3 직종 정보를 활용하지 않음**
- 개선: top3 rank 1 직종을 키로 하는 **직종별 사전 정의 자연 문장 맵**을 우선 시도. 직종 폴백도 없을 때만 radar 폴백.
- 예시:
  - 안마사 / 헬스키퍼: "사람의 몸을 부드럽게 풀어주는 손길에 강점이 있어요"
  - 점역·교정사: "글자를 꼼꼼히 살피고 정확하게 다루는 집중력이 있어요"
  - 온라인 콘텐츠 모니터링: "화면을 차분히 살피며 패턴을 찾아내는 강점이 있어요"
  - 호텔객실관리: "공간을 가지런히 정돈하는 작업에서 안정적이에요"

### 4-3. temperature 0.7 → 0.5
- `openAiFetch.ts`의 기본값을 0.5로 변경 (1차 매칭, 2차 summary 모두 적용)
- 0.3은 너무 결정적이라 표현이 굳고, 0.7은 같은 입력에도 결과가 크게 흔들림. 0.5는 따뜻한 한 문장 생성과 점수 산정 모두에 안정성·자연스러움 균형이 좋음 (OpenAI 가이드도 정확성 중시 케이스에 0.2~0.5 권장).
- 1차 점수 산정의 안정성도 함께 개선됨.

### 4-4. 화이트리스트는 유지
- 시각장애 특화 직종 등 NCS 매핑이 어려운 직종을 보존
- 추천 풀 축소로 인한 유저 손실 방지

## 5. 검토하지 않거나 미뤘던 것

- **옵션 C (1차 프롬프트에 NCS 컨텍스트 주입)**: 가장 강력하지만 토큰 비용 증가 폭이 크고, NCS 미매칭 직종은 어차피 컨텍스트 비어있음 → 효과 대비 비용이 높음. 이번 PR에서는 미적용.
- **NCS API 자체 교체/추가**: 워크넷 NCS API의 검색 정확도가 근본 원인. 다른 API(예: KEAD 직무사전) 도입은 별도 과제.
